# Task 6: Auth0 用户认证集成

> **Goal:** 实现 Web2 + Web3 混合架构的用户认证系统

## 问题背景

当前项目使用钱包地址作为用户身份，虽然符合 Web3 标准，但缺少传统 Web2 的用户体系：无法用邮箱注册/登录、无法做用户画像、会员等级等 Web2 功能。

为了高度还原混合架构设计，需要集成 Auth0。

## 📊 Web2 + Web3 认证模式对比

### 模式一：解耦模式（当前实现 ✅）

**特点**：Auth0 登录和钱包连接是独立的，用户可以选择只用其中一种

| 维度 | 说明 |
|------|------|
| 登录方式 | 用户可以用 Auth0 登录，也可以用钱包连接，也可以两者都用 |
| 权限关系 | 独立运作，互不影响 |
| 使用场景 | 公开市场（OpenSea、StepN）—— 任何人都可以浏览/交易 |

**优点**：
- 用户体验简单，不需要同时完成两个步骤
- 适合不需要知道用户真实身份的公开交易场景
- 实现简单

**缺点**：
- 无法知道链上操作对应的真实用户身份
- 无法做用户画像、会员等级等精细化运营

---

### 模式二：强制绑定模式

**特点**：只有同时满足 Auth0 登录 + 钱包连接，才能执行关键操作

| 维度 | 说明 |
|------|------|
| 登录方式 | 用户必须同时完成 Auth0 登录 AND 钱包连接 |
| 权限关系 | 必须同时满足才能执行写链上操作（预订、铸造等） |
| 使用场景 | 需要身份验证的 DApp（租房、买卖平台） |

**实现逻辑**：
```
用户点击 "预订" 按钮时：
  1. 检查 Auth0 登录状态 → 未登录？→ 提示登录
  2. 检查钱包连接状态 → 未连接？→ 提示连接钱包
  3. 两者都满足 → 执行交易
```

**优点**：
- 身份可追溯：预订的人既知道邮箱又知道钱包
- 责任明确：出问题能查到是谁
- 符合业务：租房/买房本来就需要身份验证

**缺点**：
- 用户步骤变多，转化率可能降低
- 实现稍复杂

---

### 如何选择？

| 场景 | 推荐模式 |
|------|---------|
| 公开 NFT 市场 | 解耦模式 |
| DeFi 协议 | 解耦模式 或 SIWE |
| 租房/买卖平台 | 强制绑定 |
| 需要用户画像/会员体系 | 强制绑定 |

**当前实现**：解耦模式（用户可以只连接钱包就预订）

**如果需要改成强制绑定**：在 `bookNow()` 函数中添加登录状态检查，未登录时跳转到 Auth0 登录
